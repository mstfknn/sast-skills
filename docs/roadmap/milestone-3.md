## sast-ratelimit — Missing rate limit on auth / expensive endpoints

**Framework:** API23 API4 · **Tier:** B · **CWE:** CWE-770

**Scope.** Auth and resource-intensive endpoints (login, password-reset, token-refresh, OTP
verify, search, bulk-export) that are reachable without a framework- or gateway-level rate
limit, enabling brute-force and resource-exhaustion DoS. NOT: endpoints already protected by a
global middleware that applies to all routes; endpoints protected by an upstream API gateway
(Kong, AWS API GW, Cloudflare) evidenced in IaC or config.

**Recon sinks** (recon-phase grep/AST targets):

| Language / Framework | Patterns |
|---|---|
| Express / Node | Route handlers `app.post('/login')`, `router.post` without `rateLimit(` or `slowDown(` in the middleware chain; no `express-rate-limit` import in the file or its parent router |
| Django | View functions / CBVs handling `authenticate`, `login`, `password_reset` without `django-ratelimit` `@ratelimit` decorator or `RateLimitMixin`; missing `DEFAULT_THROTTLE_CLASSES` in DRF settings |
| Spring Boot | `@PostMapping` on paths containing `login`, `token`, `otp`, `reset` without `@RateLimiter` (Resilience4j) or Bucket4j filter in the filter chain |
| FastAPI | `@router.post` on auth paths without a `Depends(RateLimiter)` or `slowapi` `@limiter.limit` decorator |
| Rails | `sessions#create`, `passwords#create` without `rack-attack` `throttle` block targeting the path |

**Verify.** Taint question: is the endpoint reachable from the public internet with no
per-IP or per-user request cap enforced at the application layer or a verified gateway?
FP-killers: a global `app.use(rateLimit({...}))` before any route registration; a
`RateLimitingFilter` / `ThrottlingFilter` applied to all requests in Spring; `Rack::Attack`
block with wildcard path; IaC evidence of an upstream gateway applying rate limits to all
traffic. `exploitability: reachable` only when no limit is found at any layer.
`confidence: high` for direct route-with-no-middleware evidence; `medium` when gateway config
is external to the repo.

**Severity.** Default **high** (auth brute-force with no back-off → credential stuffing).
Raise to **critical** when OTP or MFA verification is unprotected (trivially bypassable MFA).
Lower to **medium** when the endpoint is internal-only (non-public network boundary).

**Remediation.** Add per-IP rate limiting directly on the route or via a scoped middleware.
For auth endpoints enforce exponential back-off or account lockout after N failures. For
expensive endpoints add per-user quotas. Validate that gateway-level limits are applied before
relying on them as the sole control.

**Chains with.** `sast-missingauth` (unprotected endpoint also lacks auth check) →
`chain_id` "auth-bypass"; `sast-session` (no rate limit on login + no session regeneration
compounds brute-force impact) → `chain_id` "credential-stuffing".

**Test fixture.** TP: Express `app.post('/login', loginHandler)` with no rate-limit
middleware in scope. TN: same route preceded by `app.use('/login', rateLimit({ windowMs: 60000, max: 10 }))`. TN: Django login view with `@ratelimit(key='ip', rate='5/m', block=True)`.
Assert only the TP is flagged, `reachable`, `high`.

## sast-session — Session fixation and low-entropy session ID

**Framework:** Web25 A07 · **Tier:** B · **CWE:** CWE-384, CWE-330

**Scope.** Session IDs that are not regenerated on privilege escalation (login, sudo,
role-switch), and session tokens generated with insufficient entropy or a predictable PRNG.
NOT: stateless JWT flows (covered by `sast-jwt`); session IDs that are correctly regenerated
on every privilege change; tokens generated by a well-audited framework default that uses
CSPRNG.

**Recon sinks** (recon-phase grep/AST targets):

| Language / Framework | Patterns |
|---|---|
| Express / Node | `req.session.userId =` or `req.session.user =` (assignment) without a preceding `req.session.regenerate(` call; `session({ secret: '...' })` with a hardcoded short secret |
| Django | `login(request, user)` without `request.session.cycle_key()` call immediately before or after; `SESSION_COOKIE_SECURE = False`; custom session key generated with `random.random()` or `random.randint` |
| Spring Boot | `HttpSession.setAttribute` on auth success without `sessionManagement().sessionFixation().newSession()` or `.changeSessionId()` in `SecurityFilterChain`; `HttpSessionIdResolver` returning a deterministic ID |
| PHP | `$_SESSION['user']` assignment after `session_start()` without `session_regenerate_id(true)` |
| Rails | `session[:user_id] =` (assignment) without `reset_session` on sign-in; `SecureRandom` replaced by `rand` for token generation |

**Verify.** Taint question: does the session ID survive a login event unchanged, or is the ID
generated with a non-cryptographic source? FP-killers: `session.regenerate()` /
`session_regenerate_id(true)` / `reset_session` / `cycle_key()` called within the same
authentication handler; Spring's `sessionFixation().newSession()` configured globally;
framework-managed session IDs (e.g., `express-session` with `genid` using `crypto`).
`exploitability: reachable` when fixation is demonstrable via pre-auth cookie injection.
`confidence: high` for direct authenticate-then-assign without regenerate; `medium` for PRNG
weakness only.

**Severity.** Default **high** (session fixation → account takeover without brute force).
Raise to **critical** when the session is the only auth mechanism and the application handles
financial or medical data. Lower to **medium** when session IDs are short-lived (<5 min TTL).

**Remediation.** Call session regeneration immediately on every successful login before
setting any user attributes on the session. Use the framework's CSPRNG-backed session ID
generator; never seed with `time()`, `random()`, or user-supplied values. Set
`SESSION_COOKIE_SECURE`, `SESSION_COOKIE_HTTPONLY`, and `SameSite=Lax` (covered by
`sast-cookieflags`).

**Chains with.** `sast-cookieflags` (missing Secure/HttpOnly flags amplify session hijack
risk) → `chain_id` "session-hijack"; `sast-ratelimit` (no rate limit on login enables
fixation-and-brute) → `chain_id` "credential-stuffing".

**Test fixture.** TP: `req.session.userId = user.id` immediately after credential check with
no `req.session.regenerate()`. TN: `req.session.regenerate(() => { req.session.userId = user.id; res.json({ ok: true }); })`. TN: Django `login(request, user)` with `cycle_key()` called prior.
Assert only the TP is flagged, `reachable`, `high`.

## sast-oauth — OAuth / OIDC misconfigurations

**Framework:** API23 API2, Web25 A07 · **Tier:** B · **CWE:** CWE-601, CWE-352

**Scope.** OAuth 2.0 and OIDC implementation flaws: unvalidated `redirect_uri` (open
redirect / code-intercept), missing or unverified `state` parameter (CSRF on authorization
callback), missing PKCE for public clients, and use of the implicit grant flow. NOT: server-to-server client-credentials flows with no redirect; correctly implemented PKCE + `state`
flows; library-managed flows where the library enforces these controls by default.

**Recon sinks** (recon-phase grep/AST targets):

| Language / Framework | Patterns |
|---|---|
| Node / Passport.js | `passport.authenticate('oauth2', { ... })` callback without `state` option enabled; `redirect_uri` built from `req.query.redirect` without an allowlist check |
| Python / authlib | `oauth.create_client(...)` without `server_metadata_url` or explicit `redirect_uri` validation; callback handler reading `code` from query params without `state` comparison |
| Spring Security OAuth | `OAuth2AuthorizationRequestRedirectFilter` with custom `redirect_uri` resolver reading from request parameters; missing `pkce` in `PublicClientOAuth2AuthorizedClientProvider` |
| Django-allauth / social-auth | `SOCIAL_AUTH_ALLOWED_REDIRECT_HOSTS` not set or empty; `response_type=token` (implicit) in provider config |
| Any | `response_type=token` in authorization URL; `grant_type=implicit` in token request |

**Verify.** Taint question: can an attacker supply an arbitrary `redirect_uri` that the
authorization server accepts, or can the callback be triggered without a verified `state`?
FP-killers: `redirect_uri` compared against a hardcoded allowlist before use; `state` read
from session and compared to query param on callback; PKCE `code_verifier` generated and
stored server-side; library default that enforces `state` (e.g., `django-allauth >=0.47`).
`exploitability: reachable` when an external attacker can intercept an auth code or trigger a
CSRF authorization. `confidence: high` for explicit `redirect_uri` from request; `medium` for
missing `state` only.

**Severity.** Default **high** (auth-code interception → full account takeover).
Raise to **critical** when the authorization server issues refresh tokens with `offline_access`
scope, enabling persistent access. Lower to **medium** for missing PKCE on a confidential
(server-side) client where the client secret is the primary protection.

**Remediation.** Validate `redirect_uri` against a strict allowlist registered at the
authorization server; never accept dynamic URIs from the request. Generate a CSPRNG `state`
parameter, store in session, and compare on callback. Use PKCE (`S256`) for all public
clients. Avoid the implicit grant; use authorization-code with PKCE instead. For OIDC,
validate `nonce` in the ID token.

**Chains with.** `sast-openredirect` (unvalidated `redirect_uri` is also an open redirect)
→ `chain_id` "open-redirect-oauth"; `sast-csrf` (missing `state` is equivalent to missing
CSRF token on the callback) → `chain_id` "csrf-oauth".

**Test fixture.** TP: callback handler `const code = req.query.code; const state = req.query.state;` with no session-`state` comparison before exchanging the code.
TN: `if (req.query.state !== req.session.oauthState) return res.status(403).end();` preceding
the code exchange. TN: Passport.js config with `state: true` and library-managed comparison.
Assert only the TP is flagged, `reachable`, `high`.

## sast-routeinventory — Shadow / debug / admin routes left registered

**Framework:** API23 API9 · **Tier:** B · **CWE:** CWE-1059

**Scope.** Endpoints registered in the running application that are undocumented (not in the
OpenAPI spec), marked as deprecated but still active, or are debug / admin / health routes
accessible without auth from a public network. NOT: intentionally public health-check probes
behind an auth-exempt network policy; admin routes correctly protected behind auth middleware
and IP allowlist; routes present in the OpenAPI spec with a documented deprecation notice.

**Recon sinks** (recon-phase grep/AST targets):

| Language / Framework | Patterns |
|---|---|
| Express / Node | Route paths matching `/debug`, `/admin`, `/internal`, `/test`, `/_`, `/actuator`, `/console`; routes tagged with `// TODO`, `// deprecated`, `// remove`; `app.use('/debug', ...)` without auth middleware |
| Django | URL patterns in `urlpatterns` with names containing `debug`, `admin`, `internal`, `test` that are not in `django.contrib.admin` (separately audited); `DEBUG=True` views exposed in production URL config |
| Spring Boot | Actuator endpoints enabled via `management.endpoints.web.exposure.include=*` without `management.endpoint.*.access` restrictions; `@RequestMapping` on paths `/debug/**`, `/admin/**` without `@PreAuthorize` |
| FastAPI | `router.include_router(debug_router)` outside of `if settings.debug:` guard; endpoints decorated with `@app.get('/internal/...')` without `Depends(require_admin)` |
| Rails | `mount` or `get` in `routes.rb` with paths `/sidekiq`, `/admin`, `/letter_opener`, `/debug` without `constraints` or `before_action :authenticate_admin!` |

**Verify.** Taint question: is the route reachable unauthenticated from an external network
and absent from the published API specification? FP-killers: route is behind
`Depends(require_admin)` / `@PreAuthorize("hasRole('ADMIN')")` / `before_action
:authenticate_admin!`; route is network-restricted via `constraints(host: /internal\./)` or a
k8s NetworkPolicy documented in IaC; route is present in the OpenAPI spec with `x-internal:
true` and documented deprecation timeline. `exploitability: reachable` when the route responds
200 without credentials in a test environment. `confidence: high` for routes with debug/admin
in the path and no auth decorator; `medium` for deprecated routes still returning data.

**Severity.** Default **medium** (information disclosure, internal tooling exposed).
Raise to **high** when the shadow route allows data mutation or reveals credentials/secrets.
Raise to **critical** when a full admin console (`/rails/info`, Spring Boot `/actuator/env`,
`/sidekiq`) is unauthenticated.

**Remediation.** Remove debug and test routes before production deployment using build-time
guards (`if (process.env.NODE_ENV !== 'production')`). Protect admin routes with auth + IP
allowlist. Enable Actuator selectively (`management.endpoints.web.exposure.include=health,info`)
and secure the rest. Maintain an OpenAPI spec as the source of truth; CI should fail if a
registered route is not in the spec.

**Chains with.** `sast-missingauth` (shadow route also lacks auth) → `chain_id`
"unauthenticated-admin"; `sast-excessivedata` (debug route returns internal object data) →
`chain_id` "data-exposure-debug".

**Test fixture.** TP: `app.get('/debug/users', (req, res) => res.json(db.users.findAll()))` with
no auth middleware. TN: same route wrapped in `if (process.env.NODE_ENV !== 'production') { ... }`.
TN: `app.get('/admin/stats', authMiddleware, adminHandler)`. Assert only the TP is flagged,
`reachable`, `medium`.

## sast-unsafeconsumption — Third-party API response flowing to sink without validation

**Framework:** API23 API10 · **Tier:** B · **CWE:** CWE-20

**Scope.** Data returned from a third-party or upstream HTTP API that flows directly into a
dangerous sink (SQL query, HTML render, shell command, file path, redirect URL) without schema
validation or sanitization. NOT: data validated against a strict schema (Pydantic, Zod, JSON
Schema) before use; data from trusted internal services that share the same trust boundary;
responses parsed and only specific typed fields extracted.

**Recon sinks** (recon-phase grep/AST targets):

| Language / Framework | Patterns |
|---|---|
| Node / axios / fetch | `response.data.<field>` or `(await res.json()).<field>` flowing into template literals passed to `db.query(`, `exec(`, `res.redirect(`, `res.send(`; no Zod `parse` / `safeParse` between fetch and use |
| Python / httpx / requests | `resp.json()['field']` flowing into `cursor.execute(`, `subprocess.run(`, `render(`, `redirect(`; no Pydantic model `.model_validate(resp.json())` before field access |
| Java / RestTemplate / WebClient | `response.getBody().<getter>()` flowing into `jdbcTemplate.query(`, `Runtime.exec(`, `response.sendRedirect(`; no bean validation (`@Valid`) or manual type assertion |
| PHP / Guzzle | `$response->json()['field']` flowing into `$pdo->query(`, `shell_exec(`, `header('Location: '.)` without `filter_var` or explicit type cast |
| Ruby / Faraday | `response.body['field']` flowing into `ActiveRecord::Base.find_by_sql(`, `redirect_to(`, `system(` without explicit cast or schema check |

**Verify.** Taint question: does a field from an external API response reach a dangerous sink
without an intervening validation or type assertion step? FP-killers: a Pydantic / Zod /
JSON Schema validation call between the response and the sink; explicit type cast (`int()`,
`parseInt`) narrowing the value to a non-injectable type; response field used only in a
parameterized query or a context-safe API (e.g., ORM method accepting a typed parameter).
`exploitability: reachable` when the third-party API could be compromised or misconfigured
(supply-chain threat). `confidence: high` for direct `response.field → sink` with no
validation; `medium` when an intermediate assignment obscures the flow.

**Severity.** Default **high** (upstream compromise → second-order injection into your app).
Raise to **critical** when the sink is a shell command or file write, enabling RCE. Lower to
**medium** when the third-party service is under the same organizational control.

**Remediation.** Always validate external API responses against an explicit schema before
accessing fields (Pydantic `model_validate`, Zod `parse`, JSON Schema validation). Use typed
field extraction rather than raw dict/object access. Treat every third-party response as
untrusted input at the same level as user input. Apply the same sink-specific defenses:
parameterized queries, output encoding, redirect allowlists.

**Chains with.** `sast-sqli` (unvalidated upstream field into SQL) → `chain_id`
"upstream-sqli"; `sast-ssrf` (redirect URL taken from upstream response) → `chain_id`
"upstream-ssrf"; `sast-rce` (upstream field into shell command) → `chain_id` "upstream-rce".

**Test fixture.** TP: `const data = await fetch(thirdPartyUrl).then(r => r.json()); db.query('SELECT * FROM users WHERE id = ' + data.userId)`.
TN: `const parsed = UserSchema.parse(await fetch(thirdPartyUrl).then(r => r.json())); db.query('SELECT * FROM users WHERE id = ?', [parsed.userId])`.
Assert only the TP is flagged, `reachable`, `high`.

## sast-cloudsdk — Cloud SDK misuse: public bucket, hardcoded credential, over-broad policy

**Framework:** Web25 A02 · **Tier:** B · **CWE:** CWE-732

**Scope.** Cloud SDK calls that create or configure storage buckets with public ACLs,
inline hardcoded access keys or secrets in SDK client constructors, and IAM policies or roles
that grant `*` or wildcard resource/action permissions beyond what is needed. NOT: buckets
explicitly intended for public static-asset hosting with no sensitive data and documented as
such; credentials loaded from environment variables or secret managers at runtime; IAM
policies in test fixtures marked with `// test-only` and not deployed to production.

**Recon sinks** (recon-phase grep/AST targets):

| Cloud / SDK | Patterns |
|---|---|
| AWS (boto3 / SDK JS/Java) | `put_bucket_acl(ACL='public-read'` or `'public-read-write'`; `aws_access_key_id='AKIA...'` hardcoded; IAM policy JSON with `"Action": "*"` or `"Resource": "*"`; `BlockPublicAcls=False` in `put_public_access_block` |
| Azure (SDK) | `BlobServiceClient(account_url, credential='<key>')` with literal key; `PublicAccess.Container` or `PublicAccess.Blob` in container properties; role assignment with `RoleDefinitionId` for `Owner` or `Contributor` at subscription scope |
| GCP (google-cloud SDK) | `storage.Client()` with hardcoded `service_account_info={...}`; `bucket.make_public()`; IAM binding with `role: roles/owner` and `member: allUsers` or `allAuthenticatedUsers` |
| Terraform / CDK / Pulumi | `acl = "public-read"` on `aws_s3_bucket`; `server_side_encryption_configuration` block absent; `google_storage_bucket_iam_member` with `member = "allUsers"` |

**Verify.** Taint question: does the SDK call expose data to the public internet or grant
privileges exceeding least-privilege? FP-killers: bucket holds only public static assets with
a documented asset-type allowlist and no user-generated content; credential is read from
`os.environ`, `boto3.Session()` with no explicit key, or a secrets manager call; IAM policy
is scoped to a specific resource ARN and a minimal action set with documented justification.
`exploitability: reachable` when bucket contains user or application data.
`confidence: high` for literal `AKIA`-prefixed key or `public-read-write` ACL;
`medium` for over-broad policy without public data exposure.

**Severity.** Default **high** (data exposure or privilege escalation).
Raise to **critical** when a hardcoded credential is found (direct key compromise) or when
a `public-read-write` ACL allows arbitrary writes (data tampering / malware hosting). Lower
to **medium** when the bucket is confirmed to hold only non-sensitive public assets.

**Remediation.** Remove hardcoded credentials; use IAM instance roles, Workload Identity,
or a secrets manager. Enable `BlockPublicAcls`, `BlockPublicPolicy`, `IgnorePublicAcls`, and
`RestrictPublicBuckets` on all S3 buckets by default. Apply least-privilege IAM: scope
actions and resources to the minimum needed. Enable server-side encryption. Scan IaC with
`tfsec`, `checkov`, or `kics` in CI.

**Chains with.** `sast-hardcodedsecrets` (hardcoded key detected by secret scanner) →
`chain_id` "cloud-key-exposure"; `sast-iac` (same misconfiguration in Terraform) →
`chain_id` "iac-public-bucket".

**Test fixture.** TP: `s3.put_bucket_acl(Bucket='uploads', ACL='public-read-write')` with user-generated content bucket.
TN: `s3.put_bucket_acl(Bucket='static-assets', ACL='public-read')` with documented static-only policy.
TN: `s3.put_public_access_block(Bucket='uploads', PublicAccessBlockConfiguration={ 'BlockPublicAcls': True, ... })`.
Assert only the TP is flagged, `reachable`, `high`.

## sast-postmessage — postMessage without origin check, CSWSH, reverse-tabnabbing

**Framework:** Web25 A01 · **Tier:** B · **CWE:** CWE-346

**Scope.** Three related origin-trust failures in browser contexts: (1) `window.addEventListener('message', handler)` that processes data without verifying `event.origin`; (2) WebSocket server that does not validate the `Origin` header (cross-site WebSocket hijacking, CSWSH); (3) `<a target="_blank">` or `window.open` without `rel="noopener noreferrer"` (reverse-tabnabbing). NOT: `postMessage` handlers that unconditionally ignore or discard the event payload (no sink); WebSocket endpoints behind a CORS-equivalent allowlist; `target="_blank"` links to same-origin or trusted first-party pages with explicit `noopener`.

**Recon sinks** (recon-phase grep/AST targets):

| Context | Patterns |
|---|---|
| JavaScript (browser) | `window.addEventListener('message', fn)` or `self.addEventListener('message', fn)` with handler body that does not check `event.origin ===` or `allowedOrigins.includes(event.origin)` before using `event.data` |
| WebSocket server (Node / ws / Python / Django Channels) | Upgrade handler or `on('connection', ...)` that does not read and validate `req.headers['origin']` against an allowlist; Django Channels `WebsocketConsumer.connect()` without `self.scope['origin']` check |
| HTML / JSX / template | `href="..."` or `<a` with `target="_blank"` and no `rel="noopener noreferrer"`; `window.open(url, '_blank')` without subsequent `child.opener = null` |

**Verify.** Taint question for postMessage: does `event.data` flow to a DOM sink, `eval`,
`fetch`, or state mutation without an `event.origin` allowlist check? For CSWSH: can an
attacker initiate a WebSocket connection from a malicious page using the victim's cookies?
For tabnabbing: does the opened page have `window.opener` access to the parent?
FP-killers (postMessage): `if (event.origin !== 'https://trusted.example.com') return;` or
`if (!ALLOWED_ORIGINS.includes(event.origin)) return;` at the top of the handler.
FP-killers (CSWSH): `Origin` header compared against `settings.ALLOWED_HOSTS` or an explicit
allowlist; WebSocket endpoint is cookie-free / token-authenticated in the URL/header.
FP-killers (tabnabbing): `rel` attribute contains `noopener`; link is to a same-origin page.
`exploitability: reachable` for postMessage and CSWSH when an external page can trigger the
sink; `low` for tabnabbing (requires the linked page to be attacker-controlled).
`confidence: high` for missing origin check with data use; `medium` for tabnabbing alone.

**Severity.** Default **high** for postMessage sink-without-origin (XSS-equivalent data
injection). Default **high** for CSWSH (session/cookie-based auth bypass). Default **low**
for tabnabbing alone; raise to **medium** when the parent page handles sensitive auth flows.
Lower postMessage/CSWSH to **medium** when the application uses token auth in the WebSocket
URL (not cookies), reducing CSWSH exploitability.

**Remediation.** In `postMessage` handlers, check `event.origin` against a hardcoded
allowlist at the very start of the handler before reading `event.data`. In WebSocket servers,
validate the `Origin` header on upgrade and reject unexpected origins with a 403. Add
`rel="noopener noreferrer"` to all `target="_blank"` links or set `window.opener = null`
after `window.open`. Configure a `Content-Security-Policy` `frame-ancestors` directive to
limit embedding contexts.

**Chains with.** `sast-xss` (unvalidated `event.data` rendered into DOM) → `chain_id`
"postmessage-xss"; `sast-csrf` (CSWSH reuses CSRF-style session cookie exploit) →
`chain_id` "cswsh-csrf".

**Test fixture.** TP: `window.addEventListener('message', (e) => { document.getElementById('out').innerHTML = e.data.html; })` with no origin check.
TN: `window.addEventListener('message', (e) => { if (e.origin !== 'https://app.example.com') return; document.getElementById('out').innerHTML = e.data.html; })`.
TN (tabnabbing): `<a href="https://external.example" target="_blank" rel="noopener noreferrer">`.
Assert only the TP is flagged, `reachable`, `high`.
